## Part 2 — a full real-data wiring pass was built, then rolled back

Built and **Playwright-verified working end-to-end** (account creation →
real `profiles` row → clan join → lineage submission → real `lineages` row
→ session persists across hard reload → Bataka Panel access-code gate →
officer verifies the real submission → member's own profile shows the real
badge → clan-scoping isolation holds) — then the user asked to roll it back
to code-only-mocked. **None of what follows exists in the codebase
anymore** — recorded here so the next attempt doesn't have to
re-investigate the same design questions from scratch:

- **Real accounts without real SMS**: no SMS provider is chosen (same
  deferred-provider category as payments), so `LoginFlow`'s on-screen demo
  code stayed, but entering it called a new `POST /api/auth/phone-session`
  route that actually created a Supabase Auth user (via a deterministic
  synthetic email like `0772345678@members.ebikabyaffe.internal`, since
  Supabase requires an email/phone identifier and we're not using its real
  phone-OTP path) and minted a real session — using
  `admin.generateLink({type:"magiclink"})` immediately redeemed server-side
  via `verifyOtp({type:"magiclink", token_hash})`, so nothing is ever
  actually emailed or texted. **Working call shape, if rebuilt**:
  `verifyOtp` must be called with **only** `{ type, token_hash }` — passing
  `email` alongside `token_hash` throws `"Only the token_hash and type
  should be provided"`.
- **Session bootstrap bug found and fixed, worth remembering**: initially
  used `supabase.auth.getUser()` to check for a session on mount/reload —
  this re-verifies against Supabase's server on every call. On this
  environment's flaky network (single Supabase Auth calls ranged ~200ms to
  ~12s across identical requests), a transient hiccup on that verification
  call was indistinguishable from "not logged in" and silently signed
  people out on reload. Fix: **use `getSession()` instead**, which reads
  the locally persisted (cookie) session without a network round-trip.
  Worth checking for this exact mistake if `getUser()` shows up again in
  any future client-side session-bootstrap code.
- **A second real lint-driven fix worth remembering**: this repo's
  `react-hooks/set-state-in-effect` rule rejects a `useEffect` whose body
  synchronously calls a function that sets state (even an awaited async
  helper chained with `.finally()`) — it wants setState to happen from
  inside a genuine callback boundary. The clean fix for an auth bootstrap
  effect: don't call a separate `getSession()`-then-setState helper
  directly in the effect; instead do **only**
  `supabase.auth.onAuthStateChange(callback)` in the effect, and put all
  the profile-fetching/setState logic inside that callback — Supabase's
  documented behavior is that `onAuthStateChange` fires once immediately
  with whatever session already exists, so this one subscription handles
  both the initial-mount bootstrap and later sign-in/out, and every setState
  call is safely inside the subscription's own callback.
- **Bataka Panel had no real per-officer accounts to authorize against**
  (`panel_officers` requires a real invited `auth.uid()`, and none exist —
  the panel's only sign-in is still the "pick any clan" demo dropdown).
  Wiring that straight to real data would let anyone who finds
  `/batakaPanel` browse any clan's real submitted family lineage data,
  contradicting the privacy promise already printed on the member-facing
  form. **User chose a same-day stopgap**: a single shared passphrase
  (`BATAKA_PANEL_ACCESS_CODE`) required before the clan picker even
  appears, checked **server-side** on every panel API route (not just
  hidden in the UI) via an httpOnly cookie holding an HMAC token (keyed by
  the service-role key) rather than the passphrase itself. Real
  invitation-based officer accounts are still a good candidate for a future
  session — this stopgap is not a substitute for that.
- Panel reads/writes (with no real officer session to satisfy RLS) went
  through Route Handlers using the **service-role client**, scoped by
  whatever clan the picker session said — same trust level the demo picker
  always had, just gated instead of wide open.
- `lib/auth/registry.ts` (the session-only phone→name `Map`, s18) and
  `lib/batakaPanel/mockMembers.ts` (fictional seeded members, s17) would
  both be deleted in a real wiring pass — they were, then restored by the
  rollback along with everything else.
- The `VerificationCard.tsx` "(Demo only: simulate the Omutaka approving
  this request)" shortcut would be removed in a real wiring pass (real
  approval happens in the panel instead) — also restored by the rollback.

If picking this back up, the design decisions above (synthetic-email
session minting, `getSession()` not `getUser()`, `onAuthStateChange`-only
