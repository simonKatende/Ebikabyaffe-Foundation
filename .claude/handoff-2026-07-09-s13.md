## What was done this session (s13)

User reported two mobile-only UX problems on the **live production site**
(not local dev) and asked for both fixed:

1. Tabs required horizontal scrolling on mobile instead of a hamburger menu.
2. Tapping an image (lightbox), then pressing the phone's back button, left
   the page entirely instead of just closing the photo.

### 1. Mobile hamburger nav (`web/components/Nav.tsx`)

Below the `md` breakpoint, the old `overflow-x-auto` horizontal tab row is
now `hidden`, replaced by a hamburger icon (☰ / × toggle) that opens a
full-width dropdown listing all 7 nav tabs stacked vertically, each showing
label + sub-caption, with the same active-tab highlighting the desktop row
already had. Desktop (`md:` and up) is visually unchanged — same scrolling
tab row as before.

Closes on: clicking a link, Escape, or a click/tap outside the menu. Closes
on route change too, but **not** via the obvious `useEffect(() =>
setMenuOpen(false), [pathname])` — this repo's ESLint config has both
`react-hooks/set-state-in-effect` (blocks calling setState directly in an
effect body) and `react-hooks/refs` (blocks reading/writing a ref during
render, which rules out the usual ref-comparison "adjust state on prop
change during render" workaround too). Landed on: the hamburger button +
dropdown's `menuOpen` state was pulled out into a small `MobileMenu`
sub-component that the parent `Nav` renders with **`key={pathname}`** — a
route change gives it a fresh `key`, so React unmounts/remounts it with
`menuOpen` reset to `false` for free, no effect needed at all. If you need
this "reset local state when some parent value changes" pattern again
elsewhere in this codebase, this `key`-remount approach is the one that
satisfies this project's lint rules — don't reach for the ref-comparison
trick, it's now blocked.

### 2. Lightbox back-button fix (`web/components/ui/ImageLightbox.tsx`)

The lightbox previously only tracked open/closed as local React state in
each caller (`MmambaDetail.tsx`, `LugaveDetail.tsx`, `GenericClanDetail.tsx`
all call `setLightbox(...)`/`setLightbox(null)`) — it never touched browser
history. On mobile, pressing the system back button/gesture doesn't know
about React state; it just pops browser history, which skipped straight
past whatever page the lightbox was open on and landed the user somewhere
else entirely.

Fixed inside `ImageLightbox.tsx` itself (not the three callers) so it
applies everywhere for free: opening a photo now does
`window.history.pushState({ lightbox: true }, "")`, and a `popstate`
listener calls `onClose()`. One back press now closes the photo and keeps
the user on the same page at the same scroll position; a second back press
then does a real "leave the page" navigation, same as if the lightbox had
never been opened.

**Two real bugs found and fixed while building this** (both confirmed with
a from-scratch Playwright script against a real `npm run dev` server, not
assumed):

- **A React StrictMode double-invoke race in my own first attempt.** The
  very first version pushed history in the effect and unconditionally
  called `history.back()` in the effect's cleanup if `onClose` hadn't
  already fired. StrictMode's dev-only mount→cleanup→mount double-invoke
  triggered that `history.back()` during the *synthetic* first
  cleanup, and the resulting `popstate` event landed *async*, arriving
  after the *real* second mount had already re-attached its own listener —
  so the lightbox closed itself the instant it opened, every time. Fixed by
  guarding the push with a check on `history.state` (`historyHasLightboxEntry()`)
  instead of pushing unconditionally, and moving the `history.back()` call
  out of the effect cleanup entirely — it's now only ever triggered by a
  real user action (the × button, backdrop click, or Esc, all routed through
  one `requestClose()` helper), never by effect teardown. This is worth
  remembering for any future code that pairs `pushState` with cleanup logic
  in this codebase: StrictMode will double-invoke the effect in dev, so
  cleanup must be idempotent and must not itself trigger async side effects
  that race with the real remount.

- **A pre-existing, unrelated bug**: opening *any* lightbox already snapped
  the page's scroll position to the top the instant it opened — nothing to
  do with the back-button fix, and present before this session touched
  anything (verified with `git stash` to isolate: reproduced on the
  untouched original file too). Root cause: `web/app/globals.css` has
  `html, body { height: 100%; }`, and the lightbox's old scroll-lock
  (`document.body.style.overflow = "hidden"`) collapses body's own
  scrollable content to exactly one viewport height the moment it's
  applied — which removes all overflow at the `html` level too (since
  `html`'s scrollable extent was entirely driven by body's content height),
  forcing `window.scrollY` to clamp to 0 with nothing left to scroll.
  Switched the scroll-lock to the standard `position: fixed` + negative
  `top` offset technique (captures `scrollY`, freezes body in place
  visually without collapsing its layout box, restores + `scrollTo`s back
  on cleanup). One added subtlety: the restore on cleanup needs a forced
  synchronous reflow (`void document.body.offsetHeight`) between restoring
  `position`/`top`/`width` and calling `window.scrollTo` — without it,
  `scrollTo` can clamp against the still-stale (zero) layout extent from a
  moment earlier and silently no-op. **If any future lightbox/modal/drawer
  needs to lock body scroll in this codebase, copy this pattern — do not
  reach for `overflow: hidden` on body, it's broken by this project's own
  `height: 100%` global rule.**

Verified end-to-end with a throwaway Playwright script (mobile iPhone-13
