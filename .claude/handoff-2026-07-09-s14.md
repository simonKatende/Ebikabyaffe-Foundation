### 2. Found and fixed a self-caused mobile layout regression

Added a **fourth line of vertical content** to a hero section that already
had a documented, fragile vertical budget (see the existing "Hero section
reserves paddingBottom" and mobile-overflow gotchas in CLAUDE.md — this is
the same fragility, hit again, on the *vertical* axis this time instead of
horizontal). Screenshotting at an iPhone-13 viewport (664px visible height
after mobile browser chrome — not the nominal 844px) showed:
- The endorsement badge rendering **fully behind the fixed nav bar**
  (pushed up by the new line, badge top landed at y≈25 while the nav
  covers y:0–72).
- The 2×2 stats grid's bottom row (`18` / `12,847` tiles) **clipped off
  the bottom of the viewport entirely** — not just encroaching into the
  reserved 110px slider-caption strip, but past the hard edge of the
  `overflow-hidden` 100vh section, i.e. genuinely inaccessible.

Measured the actual numbers before guessing at a fix: total hero content
height was 635.9px against only 482px of "ideal" budget (`100vh - navH -
110px reserved`) on that viewport — a real, sizeable overflow, not a
rounding-error edge case. Fixed by making spacing/sizing **responsive**
(unprefixed = mobile, `sm:` = restores the original desktop values —
desktop is pixel-identical to before, verified via screenshot):
- Endorsement badge: smaller font/tracking/padding on mobile specifically
  so the text stays on one line instead of wrapping to two (the biggest
  single space cost), plus a smaller bottom margin.
- Kicker itself: smaller font floor, smaller bottom margin.
- Headline: smaller font floor (36px → 32px), smaller bottom margin.
- Lede paragraph: smaller font, smaller bottom margin.
- CTA button row: smaller vertical button padding, smaller bottom margin.
- Stats grid tiles: smaller vertical padding.

Brought total content height down to 537.6px on the same viewport — still
~55px into the reserved 110px strip (a soft violation, not a hard clip),
but the hard requirement (nothing invisible/cut off) is met with a bit of
margin (content bottom at 653.8px vs a 664px hard limit). Re-verified with
fresh screenshots at both the iPhone-13 mobile viewport and a 1400×900
desktop viewport — mobile now shows the full badge + kicker + headline +
lede + CTAs + all 4 stats tiles with no clipping; desktop is unchanged.

**If this hero section gets a fifth line of content in the future**, budget
for it up front rather than discovering the overflow after the fact — the
vertical margin here is now thin. The honest fix, if this becomes a
recurring problem, is probably to stop treating the hero as a fixed
100vh box with a hard-coded 110px reserved strip and either make the
reserved strip a CSS var like `--nav-h`, or let the section's height be
`min-height: 100vh` with actual content flow instead of forced centering —
neither was done here since it's a bigger change than this request asked
for.

### 3. Dev server — real gotcha hit and worth remembering

Asked to "start" the dev server. `curl localhost:3000` reported something
already listening — a leftover `node` process from *earlier in this same
session* (not some old cross-session stale server, per the last session's
handoff note about that). Tried `kill $(cat /tmp/dev.pid)` and
`pkill -f "next dev"` from the Bash tool — **neither worked**, the port
stayed occupied. Root cause: this environment's Bash tool runs under
MSYS/git-bash, which has its own PID namespace distinct from native
Windows PIDs — `$!` after backgrounding a command captures the *bash-internal*
PID, not the real `node.exe` process ID Windows knows about, so `kill`
targets a PID that doesn't correspond to the actual running process.
**Fixed by using PowerShell instead**: `Get-NetTCPConnection -LocalPort
3000 | Select-Object -ExpandProperty OwningProcess` to get the *real*
Windows PID, then `Stop-Process -Id <pid> -Force`. This is the reliable way
to kill a dev server in this environment going forward — don't trust
`kill $(cat pidfile))`/`pkill` from the Bash tool to actually land on a
`node.exe` started via `nohup ... &` from that same tool; verify via
PowerShell's `Get-NetTCPConnection` instead, especially if `curl` says the
port is still answering after a kill attempt.

Server is now running fresh at **http://localhost:3000**, confirmed
serving current code (`GET / 200` in the log right after a clean start).
