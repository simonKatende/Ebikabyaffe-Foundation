## What changed

User noticed `/login` only had a "create account" flow — no way for a returning member to sign in without retyping their name, and no detection of a phone number that's already registered. Built:

1. **One card, two modes** — `mode: "create" | "signin"` local state in `LoginFlow.tsx`, toggled by a plain button (not a `Link`, to avoid the stale-prop-on-soft-nav problem) that also calls `router.replace("/login?mode=signin", { scroll: false })` so the URL stays shareable. Same searchParams-driven-view convention as `/clans?view=bataka` — `app/login/page.tsx` reads `?mode=` server-side and passes `initialMode` to the client component, but the toggle itself doesn't rely on that prop re-syncing (App Router doesn't remount on a same-route query change).
2. **Sign-in mode** skips the names step entirely, asks only for phone, and shows "to sign in as {name}" once it recognizes the number — no re-entering names, per the user's exact request.
3. **Duplicate detection**, one direction on each side:
   - Create mode + already-registered phone → "This contact already has an account with us." + "Sign in instead →" link, **Send code button hidden**.
   - Sign-in mode + unrecognized phone → "We don't recognize that number yet." + "Create an account →" link.
4. **New file `web/lib/auth/registry.ts`** — a plain `Map<phone, name>` module store, same mental model as `lib/batakaPanel/store.ts` but simpler (no `useSyncExternalStore` needed since the reads happen during render off local `phone` state that already triggers re-renders). `registerPhone()` is called on successful create-account OTP; `getRegisteredName()`/`isPhoneRegistered()` are read during render in both modes. **Session-only, resets on hard reload** — same documented limitation as the rest of the app's mocks (`AuthContext`'s `user`, `batakaPanel`'s store). Real duplicate-detection needs the Phase 2 backend.

## Gotcha hit while verifying

**A `page.goto()` between steps resets client-only module state** (the new registry, same as it would reset `AuthContext`) — it's a full browser navigation, not an SPA transition, so the whole JS bundle reinitializes. First draft of the verification script used `page.goto()` to bounce between `/`, `/login`, and `/login?mode=signin` and the "already registered" check kept failing because the registry had been wiped each time. Fixed by driving the whole test through in-page clicks (Nav's Sign out/Sign in buttons, the mode-toggle button) so navigation stays client-side. **Any future test of session-only mock state must avoid `page.goto()` between the setup and the assertion** — use link/button clicks instead.
