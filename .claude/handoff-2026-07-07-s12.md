Three distinct, independently-requested pieces of work:

1. **`/give` renamed to "Contribution" and reworked around selectable Sacco
   options.** Nav tab label changed from "Give"/"Busuulu" to
   "Contribution"/"Support". The Monthly Busuulu tab and its whole tier flow

   `OneTimeForm`'s "Make this a monthly Busuulu" CTA removed, `Tab` type is

   box, and the submit button reads "Pay X — UGX Y →". A preset with no fixed
   amount (Donations) does *not* lock — it still shows the normal amount
   picker. `OneTimeForm` is keyed on `` `${campaign.id}:${saccoPreset?.key ?? "none"}` ``
   from the parent so switching presets remounts it with fresh state instead
   of using a `useEffect` + `setState` (which ESLint's
   `react-hooks/set-state-in-effect` correctly flagged as an anti-pattern —
   fixed by remounting instead of syncing). The same three options are now
   also clickable on the home page's Sacco section (`HomeLanding.tsx`),

5. **Found and fixed a real mobile layout bug on the home hero**, caught by
   actually rendering the live production URL at an iPhone-13 viewport
   before telling the user it was safe to share on mobile (don't skip this
   check in future — the desktop view looked completely fine, this was
   mobile-only). Root cause: the hero lede `<p>` and the 4-tile stats bar
   are both children of `flex flex-col items-center` containers. A block
   element in a non-stretching flex container with only `maxWidth` set (no
   `width`) sizes to its own **unwrapped** content width instead of the
   space actually available, then centers and overflows both edges equally
   once clipped by the hero section's `overflow-hidden` — text looked
   truncated on both left AND right simultaneously, which is the visual
   signature of this specific bug (regular viewport overflow would only cut
   one side, or trigger a horizontal scrollbar; this didn't, because the
   `overflow-hidden` section swallows it silently — `document.scrollWidth`
   checks won't catch it, you have to actually screenshot or check the
   element's own bounding box). Fixed by:
   - Lede `<p>`: added `width: "100%"` alongside the existing `maxWidth: 460`
     (`HomeLanding.tsx`) — forces it to fill/wrap within available space,
     capped at 460px on wide screens same as before.
   - Stats bar: was a non-wrapping `flex` row of 4 fixed-padding tiles
     (`px-6 py-4` each, label text forced `whitespace-nowrap`) — way wider
     than a phone screen with no way to shrink. Changed to
     `grid grid-cols-2 sm:flex w-full sm:w-auto`, responsive tile padding
     (`px-3 py-3 sm:px-6 sm:py-4`), responsive number size
     (`text-[22px] sm:text-[27px]`), removed the forced `whitespace-nowrap`
     on labels, and added a `borderBottom` for the first two tiles so the
     2×2 mobile grid still reads as one bordered panel (was only
     `borderRight` before, which only makes sense in a single row).
   - Added `cn` import to `HomeLanding.tsx` for the new conditional tile
     className (wasn't previously imported there).
   Verified with Playwright at 390px viewport (iPhone 13) on the actual
   deployed URL, not just locally — confirmed the `<p>` now resolves to
   342px (the true available width) instead of 460px, and `scrollWidth`
   matches `clientWidth`. **Redeployed after this fix** — the current live
   URL already has it, no follow-up needed unless new mobile issues surface.
   **If you touch the hero again**: any new block-level text or row of
   fixed-width items placed inside `flex flex-col items-center` (or any
   non-stretching flex container) needs an explicit `width: 100%` /
   `w-full` (or `grid`/`flex-wrap`) — `maxWidth` alone is not enough and
   this bug is easy to reintroduce silently since it's invisible on desktop.
